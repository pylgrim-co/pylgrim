# Consolidation: agent files become thin projections of the ledger

Runs only after ratification and privacy, and only when the user opts in, one explicit yes or no per file. Skipping is always fine and costs nothing: the export keeps both files' managed blocks fresh either way.

## The motion (CLAUDE.md and AGENTS.md only)

Per consented file, in this order:

1. **Ask about keep-sections first.** Show the file's headings and ask which sections, if any, should survive outside the managed block (a hand-written project intro is the common case). Everything else is about to be replaced.
2. **Archive the original, byte for byte,** to `.pylgrim/archive/<filename>.<YYYY-MM-DD>.md` (e.g. `.pylgrim/archive/CLAUDE.md.2026-07-08.md`). Copy the file exactly as it is; no reformatting, no trimming, no header. If a file with that name already exists for today, do not overwrite it; suffix `-2`, `-3`, and so on.
3. **Rewrite the live file** to: the user's keep-sections (verbatim, outside the markers), then the managed `<!-- pylgrim:begin -->` ... `<!-- pylgrim:end -->` block. Regenerate the block by running `python3 scripts/export_claudemd.py`, never by hand.
4. **Account for every original rule.** Each rule in the archived original must be one of:
   - a ratified entry (it is now in the block),
   - a proposed entry (deferred at ratification, still visible in the ledger),
   - on the cut list you showed at curation (named, not silently dropped),
   - or readable in the archive copy.

   If a rule is none of these, stop and say so before rewriting; the miss means the harvest was incomplete.

## The archive rule

The archive is the audit trail: the user can diff any consolidation and reverse it with a copy back. Therefore:

- The copy is byte-identical to the original at the moment of consolidation.
- Archive files are never edited, regenerated, or pruned by any skill.
- One archive file per consolidation event; re-consolidating later archives again with the new date.

## The never-delete rule

Originals are never deleted. Consolidation shrinks a live file to its managed block plus keep-sections; the full previous content always survives in `.pylgrim/archive/`. No motion in this skill removes an agent file from the repo, and an empty-looking result (block only) is still a present file.

## What stays manual

Tool-specific files are READ-ONLY for this skill: `.cursorrules`, `.cursor/rules/`, `.windsurfrules`, `.windsurf/rules/`, `.clinerules`, `.roorules`, `.roo/rules/`, `GEMINI.md`, `CONVENTIONS.md`, `.goosehints`, `.junie/guidelines.md`, `.github/copilot-instructions.md`, `.github/instructions/`. Their formats outrank AGENTS.md inside their own tools and their owners may rely on tool-side features the block cannot carry. The closing summary may suggest, in one line, that the user consolidate them by hand against the ratified ledger; the skill never performs it.

## Worked example

Repo has a 240-line CLAUDE.md (18 rules, 14 ratified, 2 deferred, 2 cut as platitudes) and an AGENTS.md. The user consents to consolidating CLAUDE.md, keeps its "## Local setup" section, and declines for AGENTS.md.

```
.pylgrim/archive/CLAUDE.md.2026-07-08.md   <- the 240-line original, exact bytes
CLAUDE.md                                   <- after:
    ## Local setup
    ...user's kept section, verbatim...
    <!-- pylgrim:begin -->
    ...block regenerated by export_claudemd.py...
    <!-- pylgrim:end -->
AGENTS.md                                   <- untouched body; its managed block
                                               still refreshed by the exporter
```

Closing summary line: "Consolidated CLAUDE.md (original archived to .pylgrim/archive/, 2 deferred rules remain proposed, 2 cuts were shown at curation). AGENTS.md left as is. .cursor/rules/ is tool-owned; consolidating it against the ledger is a manual step if you want it."
